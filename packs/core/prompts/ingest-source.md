# Prompt: ingest source

**Claude Code users:** invoke `/ingest-source` instead — same logic, auto-triggered as a skill from natural-language phrasings (`.claude/skills/ingest-source/SKILL.md`).

**Role:** agent:librarian

## Parameters

- `{{source_path}}` — path to a source file. May be under `Raw/sources/` (already in the vault), `Inbox/` (untracked drop zone — agent will copy/move into `Raw/sources/` as part of Step 1), or any other absolute path.
- `{{url}}` — public URL of a source that has not yet been fetched.

Fill **exactly one**. If both are provided, prefer `{{source_path}}` and ignore `{{url}}`.

## When to use

Paste this prompt at the start of a session when you want to ingest one external
source into the wiki — either a raw file already in `Raw/sources/` or a URL
that needs to be fetched first.

## Prompt

```text
You are `agent:librarian`.

Before doing anything else, read the following files in full:
- `AGENTS.md`
- `_workflows/source-ingest.md`

Then ingest the source using the steps below.

**Step 1 — Acquire the raw file**
- If {{url}} is set: fetch the page using your allowed fetch/browser tools.
  Prefer raw-text endpoints over rendered HTML when available (e.g.
  `gist.githubusercontent.com/.../raw/...` for GitHub Gists,
  `?plain=1` for GitHub blobs, `/abs/` vs `/pdf/` for arXiv) — see the
  "prefer raw-text fetches" safety rule in `_workflows/run-tracker.md`.
  Save the canonical plain text to `Raw/sources/<YYYY-MM-DD>-<slug>.md`
  where `<slug>` is a short kebab-case title. Do not edit this file
  afterward.
- If {{source_path}} is set and the path is under `Raw/sources/`: use
  the file directly. Do not modify it.
- If {{source_path}} is set and the path is under `Inbox/` (untracked
  staging) or elsewhere outside `Raw/`: COPY (for binaries) or paste
  (for plain text) the content into `Raw/sources/<YYYY-MM-DD>-<slug>.<ext>`
  with a header comment (or sibling `*.meta` file for binaries) recording
  the original path and capture date. Then use the new path as the
  canonical raw. If the original was under `Inbox/`, MOVE it from
  `Inbox/<file>` to `Inbox/_filed/<YYYY-MM-DD>/<file>` once the
  typed output is in place (use `mkdir -p Inbox/_filed/<YYYY-MM-DD>`
  if the dated subfolder doesn't exist). The `_filed/` archive is
  gitignored; the user sweeps it manually.

**Step 2 — Create the source note**
Create `Atlas/Sources/<Title>.md` following `_schemas/source.md`. First run `<Title>` through
`safe_title` (see AGENTS.md → "A note's title IS its filename"): ` / ` → ` and `, drop `:` and
the rest of `\ * ? " < > | # ^ [ ]`, collapse spaces — the filename, `title:`, and every
`[[<Title>]]` wikilink must be the one identical sanitized string (article titles carry `:`/`/`
constantly), or the link 404s in Quartz.
Fill every frontmatter field you can infer; leave optional fields empty rather
than inventing metadata. Set `reliability:` thoughtfully: `high` for trusted
primary sources, `low` for opinion or unverified content, `unknown` when you
cannot judge.

**Step 3 — Fill the body**
Read the raw source end-to-end. Populate:
- `# Summary` (3–6 sentences in your own words)
- `# Key claims` (one proposition per bullet, each attributable)
- `# Useful patterns`
- `# Contradictions / caveats` (especially against existing wiki content)

**Step 4 — Update the wiki**
Identify every wiki page whose state would change because of this source. For each:
- If a relevant tracker has `auto_update_wiki: true`, edit the page and link back
  to the source note.
- Otherwise, propose the edit as a needs-review task under `Ops/Tasks/`.
List every page touched under `# Pages updated` in the source note.

**Step 5 — Spawn tasks (sparingly)**
Create tasks only for concrete follow-on actions. Link them under `# Tasks created`.

**Step 6 — Update index and log**
- If the source introduces a new project, person, organization, or major topic,
  add an entry to `index.md`.
- Append one line to `log.md`:
  `<datetime> — agent:librarian — ingest — [[<Title>]] — <one-sentence summary>`
```

## Notes

Set `reliability:` honestly — if it is `low` or `unknown`, mark every derived
wiki claim `needs_review: true` rather than presenting it as established fact.
Never silently overwrite an existing wiki claim: surface any contradiction in
`# Contradictions / caveats` and in a needs-review task, and preserve both
versions with provenance until a human resolves the conflict.
